## 1. Обзор и Бизнес Контекст

### 1.1 Цел на документа

Настоящата спецификация дефинира функционалните и нефункционалните изисквания към AI агент, предназначен за автоматизирано генериране на информационни репорти в областта на картовите плащания по заявка на мениджърски персонал. Документът служи като основна референция за разработчиците при проектиране, имплементиране и тестване на системата.

### 1.2 Бизнес проблем

Мениджментът в сферата на картовите плащания се нуждае от бърз достъп до оперативни, финансови и регулаторни данни, разпределени в множество специализирани системи: авторизационни платформи, клиринг и сетълмент модули, scheme reporting (Visa, Mastercard), interchange системи и CRM. Изрично извън обхвата на оперативните справки са оставени тези свързани с fraud detection системи, които могат да бъдат обект на допълнителен анлиз и включване в обхвата.
Ръчното агрегиране на тези данни е бавно, отворено за грешки и изисква специализирани технически познания. Резултатът е забавяне на оперативния мениджмънт, пропуснати тенденции и реактивен вместо проактивен подход.

### 1.3 Предложено решение

Разработване на специализиран AI агент с чат-базиран интерфейс, интегриран с всички релевантни платежни системи. Агентът позволява на мениджмънта да задава въпроси и да поръчва репорти на естествен език, като получава структурирани, точни и навременни отговори с бизнес интерпретация — без да е необходима техническа компетентност за работа с изходните системи.

| Data Administrator | Конфигурира системата | Управление на конектори, RBAC, шаблони и audit log |

### 2.2 In Scope

- Чат интерфейс за промпт-базирани запитвания на естествен език (БГ / EN)
- Репорти за транзакционен обем и стойност — по карта, търговец, канал, география
- Репорти за авторизационни процеси: approval rate, decline reasons, authorization latency
- Финансови репорти: interchange income, fee income, P&L компоненти
- Клиринг и сетълмент репорти: файлови статуси, изключения, суми по нет позиция
- Scheme compliance репорти (Visa, Mastercard, Local Scheme)
- Портфейлни репорти: активни карти, нови и подновени карти, churn (напуснали клиенти), продуктови сегменти
- Насрочване на периодични репорти (daily / weekly / monthly)
- Експорт: PDF, XLSX, DOCX; интеграция с e-mail

### 2.3 Out of Scope

- Модификация на данни в изходните системи
- Авторизационни решения в реално време (real-time decisioning)
- Публикуване на данни без одобрение от CISO

## 4. Функционални Изисквания

### 4.1 Обработка на естествен език

- Поддръжка на БГ и EN; разпознаване на платежна терминология (BIN, interchange, chargeback, scheme, clearing, settlement, approval rate, fraud rate, bp)
- Multi-turn разговор с контекстна памет в сесията — drill-down въпроси без повторение на контекста
- Confidence scoring: при < 75% агентът задава уточняващи въпроси преди изпълнение
- Разпознаване на времеви изрази: вчера, тази седмица, Q1, YTD, последните 30 дни, МoM, YoY
- Разпознаване на числови прагове и сравнения: над X%, под Y bp, топ N

### 4.2 Data Connectors — Картови Системи

| Система / Конектор | Данни / Метрики |
|---|---|
| Transaction Data Warehouse | Volume, Value, Count по всички измерения |
| Authorization Platform | Approval/Decline events, Response codes, Latency |
| Clearing & Settlement | File status, Net positions, Exceptions, Value dates |
| Card Management System | Card portfolio, Issuance, Closure, Product mix, BIN data |
| Fee & Billing | Fee income, Penalty fees, Scheme fees |
| Interchange Engine | Interchange income по категория, схема, мерчант тип |
| Scheme Reporting (Visa/MC) | Compliance metrics, Scheme mandates, Penalty thresholds |

В конкретния пример на Sirma SCards всички описани конектори с изключение авторизациите следва да сочат към общата база на системата, докато за авторизационните съобщения има отделна такава.

### 4.3 Репорт Генериране

- Задължителни секции: Executive Summary (LLM-генериран нарататив) + KPI карти с RAG статус + Детайлна таблица + Визуализация (chart) + AI препоръки/интерпретация
- Поддържани визуализации: bar chart, line chart, pie chart, KPI tile, exception list
- RAG статус: автоматично изчисление спрямо конфигурируеми threshold-и по метрика
- Drill-down: всяка таблица позволява follow-up въпроси за по-детайлно ниво
- Максимално времe за генериране: 10 сек за стандартни репорти; 45 сек за сложни cross-system

### 4.4 Сигурност & Съответствие (PCI-DSS)

- Архитектурата на агента е съобразена с PCI-DSS v4.0 изисквания
- Пан данни (PAN) никога не се предават към LLM — само агрегирани и маскирани стойности
- RBAC: минимум 6 роли с row-level security по продукт, схема и организационна единица
- Пълен immutable audit log: кой е поискал какви данни, кога и какъв репорт е получил
- TLS 1.3+ за цялата комуникация; data at rest криптиране AES-256
- LLM контекстът не съдържа необработени транзакционни данни — само предварително агрегирани резултати

| Използваемост | Нов потребител с мениджърски профил генерира първи репорт без обучение за < 5 минути |

## 6. Архитектурни Ограничения

- LLM моделът получава само предварително агрегирани данни — raw транзакционни records никога не напускат Data Warehouse
- Всички data connectors са read-only и комуникират само вътре в корпоративната мрежа / private cloud tenant
- При external LLM API: задължително data residency в ЕС; DPA с провайдъра; одобрение от CISO
- Агентът не взема авторизационни решения и не модифицира данни в никоя от платежните системи
- Fraud данни с чувствителна стратегия (rules, thresholds, model features) са достъпни само за Risk роли с explicit data classification

## 7. Критерии за Приемане

### Функционални

- Агентът генерира коректен репорт при >= 90% от тест-промптовете в одобрения test suite (покриващ всички UC)
- Числовите стойности в генерираните репорти съвпадат 100% с reference стойностите от изходните системи
- RAG статусите са изчислени коректно спрямо конфигурираните threshold-и в 100% от тест случаите
- RBAC: потребител не може да достъпи данни извън своя обхват (0 unauthorized access в security tests)
- PAN данни не фигурират в нито един генериран репорт или LLM prompt (verified чрез automated scan)

### Нефункционални

- P95 latency <= 10 сек при load test с 100 едновременни потребители (стандартни репорти)
- Audit log записва 100% от достъпите без пропуски в 48-часов endurance test
- Системата постига 99.5% uptime в 4-седмичен staging период

## 9. Рискове и Митигация

| Риск | Митигация |
|---|---|
| LLM Hallucination на числа — моделът генерира неточни финансови данни | Числата се извличат директно от Data Layer; LLM генерира САМО нарататив и интерпретация. Задължителна source attribution в репорта. |
| PCI-DSS нарушение — PAN данни в LLM контекст | Архитектурна гаранция: data pipeline агрегира данните ПРЕДИ изпращане към LLM. Automated PAN detection scan на всеки prompt и response. |
| Prompt Injection — злонамерен промпт заобикаля security слоя | Input sanitization; system prompt hardening; output validation layer; rate limiting; anomaly detection на prompt patterns. |
| Неточно разпознаване на платежна терминология | Fine-tuning или RAG с корпоративен речник на платежни термини; тест suite с 200+ специализирани промпта преди prod. |
| Недостъпност на изходна система (напр. Fraud Engine) | Graceful degradation: репортът се генерира с наличните данни, ясно маркирайки липсващия компонент. Retry логика и circuit breaker. |
| Mениджментът не се доверява на AI-генерирани данни | Всеки репорт показва: source system, timestamp на данните, query изпълнен и версия на агента. Прозрачност = доверие. |

## 10. Речник на Платежни Термини

| Термин | Дефиниция |
|---|---|
| BIN (Bank Identification Number) | Първите 6-8 цифри на картата, идентифициращи издателя и продукта |
| Interchange | Такса, платима между банките при обработка на картова транзакция |
| Approval Rate | Процент успешно авторизирани транзакции от всички опити |
| Fraud Rate (bp) | Стойност на измамни транзакции / обща стойност в базисни точки (1 bp = 0.01%) |
| Chargeback | Оспорена транзакция, върната от картодържателя чрез банката му |
| Clearing | Процес на обмен и съгласуване на транзакционни файлове между банки и scheme |
| Settlement | Финансово уреждане (превод на средства) след клиринг |
| 3DS2 (3D Secure 2) | Протокол за автентикация при онлайн транзакции (EMV 3-D Secure) |
| CNP (Card Not Present) | Транзакция без физическо присъствие на картата (e-commerce, MOTO) |
| ATO (Account Takeover) | Измама чрез завземане на клиентски акаунт/карта от трета страна |
| RAG Status | Red / Amber / Green — цветови индикатор за отклонение от целеви стойности |
| Net Position | Нетната сума за получаване/изплащане след клиринг между две страни |
| Scheme | Картова мрежа/организация: Visa, Mastercard, BCARD и др. |
| PCI-DSS | Payment Card Industry Data Security Standard — стандарт за защита на карти |
